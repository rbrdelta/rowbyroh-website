Infrastructure Beats Convenience: Vault vs. Memory

Why I built a 672-note vault instead of relying on Claude's built-in memory.

I spent an hour arguing with Claude about whether my own knowledge vault was worth building. Claude has built-in memory now — it remembers who I am, what I'm working on, my preferences. Why build an entire Obsidian MCP system when memory already works?

The answer came from the conversation itself: I was having the debate on claude.ai, where Claude couldn't access my vault. It was reasoning about my infrastructure from a "stale, incomplete mental model" — its own words. That's the gap. Memory knows that I'm building something. The vault knows what I've built, why, and what's next.

The four things a vault does that memory can't

Structure as thought. Memory is a flat summary. The vault has backlinks, tag topology, relationship emergence. The way I organize notes is a thinking process. Memory flattens that into prose.

Durable artifacts. My competitive analyses, architecture docs, financial models — these are living reference documents. Memory knows they exist. The MCP connection lets Claude actually work with them.

Serendipitous retrieval. Backlinks surface connections I didn't ask for. Memory gives you what the system decided was important. The vault gives you what you decided was important, plus what the structure implies.

Source of truth you control. If you're building toward something, you need a canonical store that doesn't depend on a third party's summarization layer. Memory is a convenience; the vault is infrastructure.

Where memory actually wins

It's not zero. Memory handles conversational continuity without friction — you don't preload context every time. It handles implicit personalization: tone, depth calibration. And cross-session coherence on low-stakes stuff like family logistics and learning paths.

Memory is optimized for "who is Daniel." The vault is optimized for "what is the current state of Daniel's projects." Those are fundamentally different retrieval problems.

The ownership argument

Even if Claude ships structured memory, graph-based recall, or vault-like persistence — the fundamental question remains: do you want your intellectual capital inside someone else's system, or inside one you control?

Memory is curated by an external process. You don't control the ontology. You don't decide what's signal vs. noise. You can't export it, version it, back it up, or build tooling on top of it.

The vault is mine. It syncs to my phone via Obsidian Sync. It's searchable from Claude Code via MCP. It's classified by a taxonomy I designed. The links are generated by an algorithm I wrote. If Anthropic changes their memory system tomorrow, my vault still works.

The test that proved the point

During the conversation, I deliberately withheld context to see what Claude could retrieve from memory alone. It could tell me I was building a PKM system. It couldn't tell me what phase it was in, what was working, what was broken, or what we'd decided in the last session. It was "speculating at best" — Claude's own admission.

Then I opened Claude Code, where the MCP connection was live. Same question. Full grounded context in seconds.

That's the difference between convenience and infrastructure.

The principle

Build the infrastructure layer even when the convenience layer seems good enough. The convenience layer is optimized for the common case. The infrastructure layer is optimized for your case. And over time, the compound value of infrastructure always exceeds the immediate comfort of convenience.
